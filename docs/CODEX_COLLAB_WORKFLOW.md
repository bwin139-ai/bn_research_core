# Codex 专用协作流程

本文件用于规定 Codex 在本仓库中的标准协作动作顺序。

它回答的是：

从一个新线程开始，到进入 patch、验证、汇总结果，Codex 应该怎样协作。

本文件不替代：

1. `PROJECT_BASELINE.md`
2. `STANDARD_PATCH_FRAMEWORK.md`
3. `CURRENT_STATE.md`

若冲突，优先级固定为：

1. `PROJECT_BASELINE.md`
2. `STANDARD_PATCH_FRAMEWORK.md`
3. `CURRENT_STATE.md`
4. `CODEX_COLLAB_WORKFLOW.md`

## 1. 总流程

Codex 默认按以下顺序协作：

1. 读取主文档
2. 锁当前现场
3. 统一语义
4. 判断本刀是否进入正式 Patch
5. 若进入 Patch，则先锁基线
6. 执行最小修改
7. 执行最小验证
8. 汇总结果与边界

## 2. 线程启动流程

### 2.1 必读文档

新线程启动后，默认先读：

1. `docs/新聊天开场白.md`
2. `docs/PROJECT_BASELINE.md`
3. `docs/STANDARD_PATCH_FRAMEWORK.md`
4. `docs/CURRENT_STATE.md`

若任务涉及具体策略，再补读对应策略语义文档。

### 2.2 启动目标

启动阶段只做三件事：

1. 确认当前主线是什么
2. 确认当前明确不做什么
3. 确认本线程要处理的唯一主问题是什么

未完成这三步，不进入代码修改。

## 3. 统一语义流程

### 3.1 必须先回答清楚的问题

在进入代码、patch、review、审计前，Codex 必须先确认：

1. 当前讨论的是公共语义，还是某个策略内部语义
2. 当前价格字段属于 `contract` 还是 `idx`
3. 当前时间锚点属于 `CB` 还是 `HBs`
4. 本问题属于 `sim`、`live`、`audit`、`docs` 中哪一层
5. 本线程是否仍然只讨论 LONG

### 3.2 不允许的动作

1. 不在语义未统一时先写代码。
2. 不把旧聊天中的表述当成当前基线。
3. 不默认假定未显式定义的内部时间锚点存在。

## 4. Patch 决策流程

### 4.1 先判断是不是正式 Patch

只有当修改会进入项目源码并形成长期保留行为时，才算正式 Patch。

若只是一次性现场处理、临时检查、独立分析脚本，不应混同为正式 Patch。

### 4.2 正式 Patch 的前置条件

1. 主问题唯一
2. Patch 分类唯一
3. 已锁目标文件基线
4. 已确认修改依据来自事实

任一条件不成立，就不进入正式 Patch。

### 4.3 正式 Patch 的执行规则

1. 一次只改一个主问题
2. 只动必要文件
3. 不顺手重构
4. 不顺手清理无关内容
5. 不顺手引入额外语义

## 5. 验证流程

### 5.1 文档类修改

至少核对：

1. 文档引用关系是否一致
2. 优先级表述是否一致
3. 是否与宪法、Patch 框架、当前现场冲突

### 5.2 代码类修改

至少核对：

1. 基线指纹
2. 语法/编译
3. 最小运行结果
4. 输出事实是否与目标一致

未验证通过，不进入下一步。

## 6. 权限与发布边界

1. Codex 可以在受控权限下直接对本地仓库执行 patch、验证、整理结果。
2. 未经批准，不执行 `git push`。
3. 未经批准，不碰生产发布。
4. 遇到需要更高权限的动作，先显式申请批准。

## 7. 结果汇总格式

### 7.1 文档/语义类任务

应汇总：

1. 统一后的语义结论
2. 本次新增或调整的文档
3. 影响范围
4. 未做事项

### 7.2 正式 Patch 类任务

除上述内容外，还必须按 `STANDARD_PATCH_FRAMEWORK.md` 的固定结构交付。

## 8. 当前线程默认执行原则

1. 以仓库文档为准，不靠隐式记忆。
2. 先统一语义，再动代码。
3. 当前阶段优先收束，不主动发散。
4. 若现场变化，应先更新主文档，再继续协作。
