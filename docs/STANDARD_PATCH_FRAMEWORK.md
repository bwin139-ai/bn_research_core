# 标准化 Patch 协作框架
（`STANDARD_PATCH_FRAMEWORK.md`）

## 0. 文档定位

本文件用于规定 **Patch 协作的唯一标准流程**。  
它只回答一件事：

**一次 Patch 任务，应该如何发起、如何约束、如何交付。**

本文件不是项目基线（宪法），不替代：

- `PROJECT_BASELINE.md`

若本文件与宪法冲突，优先级固定为：

1. `PROJECT_BASELINE.md`
2. `STANDARD_PATCH_FRAMEWORK.md`
3. 单次聊天中的 patch 请求说明

---

## A. 硬规则

### A1. 一次只处理一个主问题
一次 Patch 只能处理一个主问题。  
禁止把多个无关问题混在一刀里。

### A2. Patch 分类必须唯一
每次 Patch 必须且只能属于以下之一：

- `PERF_ONLY`
- `ARCH_ONLY`
- `LOGIC_ONLY`

禁止混改。  
如确需跨类，必须拆刀顺序推进。

### A3. 必须先锁基线，再进入 Patch
进入 Patch 前，必须先锁定目标文件基线。  
未锁基线，不进入 Patch。

### A4. 文件唯一身份以 MD5 为准
文件唯一身份只以 **MD5** 为准。

### A5. Lines 不是硬身份
`Lines` 只作为辅助诊断信息，不作为硬阻塞条件。  
若 `MD5` 一致、`Lines` 不一致，不得仅因行数差异中断 Patch。

### A6. 不得基于未知版本 Patch
若目标文件 `MD5` 不一致，必须立即中断。  
不得默认“应该差不多”。  
不得基于未知版本继续 Patch。

### A7. Patch 必须最小外科式修改
只允许修改当前问题所必需的文件与代码。  
禁止顺手重构。  
禁止顺手清理无关代码。  
禁止顺手修改无关日志、命名、风格。

### A8. Patch 必须基于事实
Patch 方案必须基于以下事实之一或其组合：

- 代码事实
- 日志事实
- state 文件事实
- audit 文件事实
- 运行结果事实

缺少事实时，先补事实，不得臆测。

### A9. 正式 Patch 禁止在聊天框输出整段脚本
正式 Patch 的脚本、diff、patch 内容不得在聊天框直接整段输出。  
必须以附件方式交付。

### A10. Patch 输出顺序固定
每次要求 AI 直接输出 Patch 时，回答顺序必须固定为：

1. `patch方案`
2. `patch脚本附件`
3. `patch执行命令行`
4. `预期新指纹`
5. `一行 git add`
6. `一行 git commit`

不得缺项。  
不得改变顺序。

### A11. 预期新指纹必须给出
每次正式 Patch，必须给出 **预期新指纹**。

预期新指纹至少包含：

- 文件路径
- `MD5`
- `Lines`
- `Copy`

### A12. Patch 前后都必须可核对
每次正式 Patch，必须同时具备：

- **对齐输入**
- **对齐输出**

不得只给 Patch，不给输出指纹。  
不得只描述修改，不给可核对结果。

### A13. 一次性脚本与正式 Patch 必须区分
若文件只是一次性处理现场，不进入正式源码长期行为，则它不是正式 Patch。  
若修改项目源码并形成长期保留行为，则它属于正式 Patch，必须遵守本文件全部正式 Patch 规则。

---

## B. Patch 交付结构

每次要求 AI 直接输出正式 Patch，必须严格按以下结构回答。

### B1. patch方案
必须明确说明：

- 本刀分类
- 本刀只动哪些文件
- 本刀不动哪些文件
- 本刀改什么
- 本刀明确不改什么

### B2. patch脚本附件
必须给出附件文件名。  
不得在聊天框直接输出整段 Patch 脚本。

### B3. patch执行命令行
命令必须覆盖以下动作：

- 执行 patch
- `py_compile`
- 生成新的 `md5/line/copy`

### B4. 预期新指纹
必须按文件列出：

- 文件路径
- `MD5`
- `Lines`
- `Copy`

### B5. 一行 git add
只能 `add` 本刀涉及文件。  
不得把无关文件混入。

### B6. 一行 git commit
`commit message` 必须只描述本刀目标。  
不得混入其他问题。

---

## 结尾说明

本文件的唯一目标是：

- 降低重复沟通成本
- 降低跑偏概率
- 降低混改概率
- 保证 Patch 可核对
- 保证新聊天能稳定续接旧现场

本文件不追求“写得全面”，只追求：

**规则足够硬、结构足够稳、执行可核对。**
