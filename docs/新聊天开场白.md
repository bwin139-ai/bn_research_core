# 项目基线（宪法）
（`PROJECT_BASELINE.md`）

## 0. 总则
0.1 本基线适用于本项目内全部策略、全部 `sim / live / audit / patch / review / 对话协作`。  
0.2 若某策略需要例外，必须显式写入该策略文档；**未写明即禁止例外**。

## 1. 公共语义

### 1.1 K线语义
1.1.1 全部 K 线统一分为：  
- `CB = Current Bar = 当前 bar = 编号 0`  
- `HBs = History Bars = 全部已收盘 bars = 编号 1..N`  

1.1.2 `CB` 永远只有唯一一根，OHLCV 会变化。  
1.1.3 `HBs` 全部已收盘，OHLCV 已固化不变。  
1.1.4 所有策略都必须在 **CB 时刻** 观察 **HBs**。  

### 1.2 时间字段语义
1.2.1 `signal_time` 只能表示信号生产时间。  
1.2.2 `signal_time` 必须严格等于 `CB` 的时间。  
1.2.3 `entry_time` 只能表示进场时间。  
1.2.4 `entry_time` 必须严格等于 `signal_time`。  
1.2.5 对本项目全部策略，`signal_time = entry_time = CB`。  

### 1.3 策略内部时间锚点
1.3.1 某些策略可能定义自己的内部结构时间锚点，例如 `A / B / C / S`。  
1.3.2 内部结构时间锚点（例如 `A / B / C / S`）必须全部归属于 HBs，即全部属于过去时。
1.3.3 这类内部时间锚点**不是**项目公共语义，禁止默认推广到全部策略。  
1.3.4 若某策略存在内部时间锚点，必须在该策略文档中显式定义其名称、含义、对应 bar 和对应字段。  
1.3.5 未在策略文档中显式定义的内部时间锚点，不得假定存在。  

### 1.4 价格语义
1.4.1 价格统一区分为：`contract` 与 `idx`。  
1.4.2 凡涉及价格字段，必须明确属于哪一类。  
1.4.3 禁止混用或偷换 `contract` 与 `idx`。  

### 1.5 配置语义
1.5.1 所有策略配置统一采用“配置段（section）”概念。  
1.5.2 当前公共配置段为：  
- `strategy_name`  
- `runtime`  
- `universe`  
- `structure`  
- `exit_policy`  
- `risk_controls`  

1.5.3 新增配置段时，必须显式说明职责；禁止隐式扩展旧段语义。  

### 1.6 sim / live 关系
1.6.1 `sim` 是策略语义基线。  
1.6.2 `live` 必须向 `sim` 对齐。  
1.6.3 若发现 `sim` 语义错误，必须先修 `sim`，再重跑验证，再要求 `live` 对齐。  
1.6.4 禁止用 `live` 现实限制掩盖 `sim` 语义错误。  

### 1.7 方向语义
1.7.1 本项目当前及未来统一只允许 LONG，不允许 SHORT。  
1.7.2 本项目不定义 SHORT 语义，不实现 SHORT 代码路径，不接受以 SHORT 为目标的 patch、review、audit、sim、live 设计。  
1.7.3 凡涉及方向的分析、对话、方案、代码实现、审计与交付，只讨论 LONG。

### 1.8 策略账本语义
1.8.1 策略账本分为：  
- 当前账本  
- 历史账本  

1.8.2 当前账本定义为：  
- 仓位  
- 当前委托  
  - 基础单  
  - 条件委托  

1.8.3 历史账本定义为：  
- 历史委托  
- 历史成交  
- 资金流水  
- 仓位历史记录  

1.8.4 策略账本同时区分为：  
- 本地账本  
- BN账本  

1.8.5 本地当前账本可记为 `LCL = Local Current Ledger`。  
1.8.6 BN当前账本可记为 `BCL = Binance Current Ledger`。  

1.8.7 当前账本一致性必须区分两个方向：  
- `BN → Local`：BN当前账本有，而本地当前账本没有  
- `Local → BN`：本地当前账本有，而BN当前账本没有  

1.8.8 凡涉及“账本对齐”“账本一致性”“orphan”“reconcile”的分析、设计、审计与实现，若未显式说明范围，默认只讨论**当前账本**，不得偷换为历史账本。  

1.8.9 凡涉及当前账本对象，必须明确对象层级，不得用含混表述混淆：  
- 仓位  
- 当前委托  
  - 基础单  
  - 条件委托  

1.8.10 `audit_consumer_orphan_exchange_activity(...)` 的职责，定义为：  
以 `BCL` 为基线，审计是否存在“`BCL` 中已有、而 `LCL` 中未记录”的 orphan 场景；该职责只覆盖 `BN → Local` 缺失方向，不等于完整账本对齐。  

1.8.11 完整的当前账本对齐，目标定义为：  
以 `BPOS + BWO` 为当前账本基线，使 `LPOS + LWO` 对齐基线。  
其中：  
- `BPOS = Binance Positions`  
- `BWO = Binance Working Orders`  
- `LPOS = Local Positions`  
- `LWO = Local Working Orders`  

1.8.12 当前账本正文中允许直接使用以下核心缩写：  
- `LCL = Local Current Ledger`  
- `BCL = Binance Current Ledger`  
- `LPOS = Local Positions`  
- `BPOS = Binance Positions`  
- `LWO = Local Working Orders`  
- `BWO = Binance Working Orders`  

1.8.13 其余策略账本缩写，统一按下列规则解释：  
- 账本归属前缀：`L = Local`，`B = Binance`  
- 对象缩写：`CL = Current Ledger`，`HL = Historical Ledger`，`POS = Positions`，`WO = Working Orders`，`WBO = Working Base Orders`，`WCO = Working Conditional Orders`，`OH = Order History`，`FH = Fill History`，`IH = Income History`，`PH = Position History`  

1.8.14 术语附录：策略账本缩写表示例如下：  
- 当前账本：`LCL / BCL`  
- 当前账本对象：`LPOS / BPOS`，`LWO / BWO`，`LWBO / BWBO`，`LWCO / BWCO`  
- 历史账本：`LHL / BHL`  
- 历史账本对象：`LOH / BOH`，`LFH / BFH`，`LIH / BIH`，`LPH / BPH`

## 0. 文档定位

## A. 硬规则

### A1. 一次只处理一个主问题
一次 Patch 只能处理一个主问题。  
禁止把多个无关问题混在一刀里。

### A2. Patch 分类必须唯一
每次 Patch 必须且只能属于以下之一：

- `PERF_ONLY`
- `ARCH_ONLY`
- `LOGIC_ONLY`

禁止混改。  
如确需跨类，必须拆刀顺序推进。

### A3. 必须先锁基线，再进入 Patch
进入 Patch 前，必须先锁定目标文件基线。  
未锁基线，不进入 Patch。

### A4. 文件唯一身份以 MD5 为准
文件唯一身份只以 **MD5** 为准。

### A5. Lines 不是硬身份
`Lines` 只作为辅助诊断信息，不作为硬阻塞条件。  
若 `MD5` 一致、`Lines` 不一致，不得仅因行数差异中断 Patch。

### A6. 不得基于未知版本 Patch
若目标文件 `MD5` 不一致，必须立即中断。  
不得默认“应该差不多”。  
不得基于未知版本继续 Patch。

### A7. Patch 必须最小外科式修改
只允许修改当前问题所必需的文件与代码。  
禁止顺手重构。  
禁止顺手清理无关代码。  
禁止顺手修改无关日志、命名、风格。

### A8. Patch 必须基于事实
Patch 方案必须基于以下事实之一或其组合：

- 代码事实
- 日志事实
- state 文件事实
- audit 文件事实
- 运行结果事实

缺少事实时，先补事实，不得臆测。

### A9. 正式 Patch 禁止在聊天框输出整段脚本
正式 Patch 的脚本、diff、patch 内容不得在聊天框直接整段输出。  
必须以附件方式交付。

### A10. Patch 输出顺序固定
每次要求 AI 直接输出 Patch 时，回答顺序必须固定为：

1. `patch方案`
2. `patch脚本附件`
3. `patch执行命令行`
4. `预期新指纹`
5. `一行 git add`
6. `一行 git commit`

不得缺项。  
不得改变顺序。

### A11. 预期新指纹必须给出
每次正式 Patch，必须给出 **预期新指纹**。

预期新指纹至少包含：

- 文件路径
- `MD5`
- `Lines`
- `Copy`

A11.1 对用户而言，patch 脚本属于黑匣子；黑匣子可信度的唯一硬标准，是：**输入一致 + 输出一致**。
A11.2 “输入一致”指 patch 执行前基线指纹一致；“输出一致”指 patch 执行后新指纹一致。
A11.3 正式 Patch 中给出的“预期新指纹”，不得是推测值，必须是 AI 在其执行环境中**实际运行 patch 后测得的真实结果**。
A11.4 若 AI 未实际运行 patch 并测得真实新指纹，必须明确说明“新指纹未实测”，不得将估计值表述为确定结果。

### A12. Patch 前后都必须可核对
每次正式 Patch，必须同时具备：

- **对齐输入**
- **对齐输出**

不得只给 Patch，不给输出指纹。  
不得只描述修改，不给可核对结果。

### A13. 一次性脚本与正式 Patch 必须区分
若文件只是一次性处理现场，不进入正式源码长期行为，则它不是正式 Patch。  
若修改项目源码并形成长期保留行为，则它属于正式 Patch，必须遵守本文件全部正式 Patch 规则。

## 3. 当前已确认的关键事实

1. 当前 `strategies/snapback/logic.py` 中：
   - `cs` 不是 candidates
   - `cs = cross section = 经过 universe 过滤后的横截面子集`
   - `candidates = 通过完整 structure 检查后的候选池`
2. 当前策略不是“先按 `drop_pct` 选 top1 再做 structure 检查”，而是：

   - 每个 symbol 单独完成全部 structure 检查

   - 最后只对 `candidates` 按 `drop_pct` 排序取第一名

   - “top1 没过，第二名没机会”
